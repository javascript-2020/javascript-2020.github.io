x509 Certificates in Javascript : Encrypt / Decrypt Data

26 Apr 2026

This blog post describes using x509 certificates to encrypt and decrypt data in javascript environments.
This can be useful since the public certificate can be made available publicly, even sent via an insecure communication channel, this public certificate can be used to encrypt data, the encrypted data can be sent to the holder of the private key, who can subsequently decode the encrypted data using the private key.
If two parties want to have a bi-directional encrypted communication, they can both send each other their public certificates.

The procedure uses RSA encryption to encrypt an AES symmetric key. The AES symmetric key and encryption parameters are then sent to the receiver with the encrypted payload. The sender uses the receivers RSA public key to encrypt the AES symmetric key. The receiver uses the RSA private key to decrypt the AES symmetric key and together with the encryption parameters decrypts the payload. key
The procedure is used because, AES symmetric key encryption is quick and easy. The mathematics of RSA encryption only allows for small amount of information to be encrypted.

  • 2048 bit RSA key => ~ 190 Bytes of encrypted payload
  • 4096 bit RSA key => ~ 340 Bytes of encrypted payload

notes :

private keys :

  • Browsers only support PKCS#8
  • Node.js supports both PKCS#1 and PKCS#8 private keys

certificates :

  • Browsers require the spki component in der form
  • node.js supports the entire certificate in pem format

So the rule of thumb:

  • If you want browser compatibility, always use PKCS#8.
  • If you’re in Node, you can use either.

The Secure Workflow

The correct steps for sending a secure payload using x509 private key and certificate:

  1. Receiver generates RSA Private Key & X.509 Certificate (with Key Encipherment allowed).

  2. Receiver sends the X.509 Certificate to the Sender.

  3. Sender generates a random AES-GCM key and a 96-bit IV.

  4. Sender encrypts the payload with the AES key and IV.

  5. Sender encrypts the AES key using the Receiver's RSA Public Key (from the cert).

  6. Sender sends three things to the Receiver:

    • The encrypted payload.

    • The 96-bit IV (unencrypted, this is safe to expose).

    • The RSA-encrypted AES key.

  7. Receiver uses their RSA Private Key to decrypt the AES key.

  8. Receiver uses the decrypted AES key + the IV to decrypt the payload.

PKCS#8

PKCS#8 is a standard format for storing private keys, regardless of the algorithm (RSA, EC, Ed25519, etc.). It’s defined in RFC 5208 and RFC 5958.
Think of it as:
A universal container format for private keys.
It wraps the private key with metadata describing:

  • the algorithm (RSA, EC, etc.)
  • parameters (curve, padding, etc.)
  • the key material itself

This makes PKCS#8 algorithm‑agnostic, unlike PKCS#1.

PKCS#1 vs PKCS#8 (the important difference)

PKCS#1

  • Only for RSA keys
  • Contains just the RSA private key integers
  • PEM header: -----BEGIN RSA PRIVATE KEY-----

PKCS#8

  • Works for any key type (RSA, EC, Ed25519, etc.)
  • Contains:
    • algorithm identifier
    • private key data
  • PEM headers : -----BEGIN PRIVATE KEY----- -----BEGIN ENCRYPTED PRIVATE KEY-----

Why PKCS#8 is preferred today

  • It’s universal
  • It’s compatible with modern crypto APIs
  • It supports encryption natively
  • It’s required by WebCrypto

Why Node.js supports both PKCS#1 and PKCS#8

Node’s crypto module uses OpenSSL, which understands both formats. Node doesn’t care — OpenSSL parses it.

Why browsers only support PKCS#8

WebCrypto is intentionally strict and minimal.
It only accepts:

  • PKCS#8 for private keys
  • SPKI for public keys
  • JWK for JSON keys

Converting PKCS#1 → PKCS#8 (OpenSSL)

If you have ( pcks#1 ) : -----BEGIN RSA PRIVATE KEY----- Convert it for use in browsers :

openssl pkcs8 -topk8 -nocrypt -in pkcs1.pem -out pkcs8.pem

SPKI

SPKI in X.509 stands for Subject Public Key Info.
It is the part of an X.509 certificate that contains the subject’s public key and information about the algorithm used with that key.

What SPKI actually contains

The Subject Public Key Info (SPKI) field is a structured ASN.1 block inside every X.509 certificate. It includes:

  • The subject’s public key (e.g., RSA, ECDSA, Ed25519)
  • The algorithm identifier for that key (e.g., rsaEncryption, id-ecPublicKey)
  • Any algorithm parameters required by that key type

This is the part of the certificate that clients use to verify signatures or establish encrypted sessions.

Why SPKI matters

SPKI is essential because:

  • It is the actual public key that the certificate binds to an identity.
  • It is used by TLS, code signing, S/MIME, and other PKI systems to verify authenticity.
  • It is the part that gets hashed when generating a certificate pin (HPKP-style pins or modern SPKI pinning in some systems).

In short, SPKI is the cryptographic heart of the certificate.

Where SPKI sits in the certificate structure

According to the X.509 standard, SPKI is one of the core Version 1 certificate fields, meaning it has existed since the earliest form of the standard.

Typical X.509 structure (simplified):

  • Version
  • Serial Number
  • Signature Algorithm
  • Issuer
  • Validity
  • Subject
  • Subject Public Key Info (SPKI)
  • Extensions (v3)
  • Signature

what size rsa keys do i need to encrypt an aes key of 256 bits

To safely encrypt a 256-bit AES key using RSA, you should use an RSA key size of at least 2048 bits, though 3072 bits or 4096 bits is highly recommended for modern applications requiring long-term security.

The reason for this requires looking at how RSA and AES work together, as well as the strict physical limits of RSA encryption.

In cryptography, you want the security strength of your asymmetric key (RSA) to match or exceed the security strength of your symmetric key (AES).

2048-bit RSA provides roughly 112 bits of effective security strength (vulnerable to well-funded adversaries or long-term storage cracking over the next decade).

3072-bit RSA provides roughly 128 bits of effective security strength (the current NIST recommended minimum for modern systems).

4096-bit RSA provides roughly 140+ bits of effective security strength.

While mathematically a 3072-bit or 4096-bit RSA key is technically required to get closer to the massive security margin of a 256-bit AES key, 2048-bit is the absolute bare minimum allowed by modern cryptographic standards.

Padding Overhead

You can easily fit a 256-bit (32-byte) AES key into even the smallest standard RSA key size because of how RSA handles data.

An RSA key can only encrypt data that is smaller than its own modulus size, minus the required bytes for secure padding (like OAEP).

RSA Key Size Max Total Capacity Padding Overhead (OAEP SHA-256) Max Allowed Payload Space Can it fit a 32-byte AES key?
1024-bit (Obsolete) 128 bytes 66 bytes 62 bytes Yes (But insecure)
2048-bit 256 bytes 66 bytes 190 bytes Yes
3072-bit 384 bytes 66 bytes 318 bytes Yes
4096-bit 512 bytes 66 bytes 446 bytes Yes

Even a 2048-bit RSA key gives you 190 bytes of payload space, which is more than enough room to slot your 32-byte AES key into.

Recommendation

Go with 3072-bit if you want the optimal balance between tight security and fast performance, or 4096-bit if execution time isn't a bottleneck in your application and you want maximum security. Avoid 2048-bit if you are designing a brand-new system from scratch.

how does iv length affect things? whats the deal with 96 bits

When dealing with encryption (specifically symmetric encryption like AES), the Initialization Vector (IV) ensures that encrypting the exact same data twice results in two completely different ciphertexts. This prevents attackers from spotting patterns in your data.

How the IV length affects things depends heavily on the encryption mode you are using. The "96 bits" sweet spot is tied directly to the most popular modern encryption mode: AES-GCM.

Here is the breakdown of why IV length matters and what makes 96 bits special.

Why 96 Bits (12 Bytes) is the Golden Number

The 96-bit rule is specific to AES-GCM (Galois/Counter Mode), which is the industry standard for authenticated encryption.

When you use AES-GCM, the internal mechanism requires a 128-bit block to work with. It splits that 128-bit block into two parts:

  • The IV : 96 bits.

  • A Counter: 32 bits (which starts at 0000...0001 and increments for every block of data encrypted).

AES-GCM Internal Block (128 bits total)
                  +------------------------------------+------------------+
                  |           IV / Nonce               |     Counter      |
                  |            (96 bits)               |    (32 bits)     |
                  +------------------------------------+------------------+

What happens if you use a 96-bit IV?

If you supply exactly 96 bits, the cryptographic engine simply pads it with a 32-bit counter starting at 1. This is direct, incredibly fast, and requires zero extra computational overhead.

What happens if you use any other size (e.g., 128 bits)?

If you pass an IV that is not 96 bits, the algorithm cannot use it directly. Instead, it has to run your IV through a hashing function called GHASH to mathematically compress or stretch it into a 96-bit value first.

The Downside: This adds extra computation (hurting performance), can subtly introduce a higher risk of IV collisions, and can lead to implementation bugs across different programming languages.

Rule of Thumb for GCM: Always use a 96-bit (12-byte) IV for AES-GCM. Using anything else is less efficient and prone to compatibility headaches.

What About Other Modes (Like AES-CBC)?

If you are using an older mode like AES-CBC (Cipher Block Chaining), the rule is different.

CBC mode requires the IV to be exactly the same size as the cipher's block size. Since AES always has a block size of 128 bits, AES-CBC requires a 128-bit (16-byte) IV.

If you give it a 96-bit IV, the code will throw an error or crash because it lacks the bytes needed to scramble the first block of data.

How IV Length Affects Security (The "Birthday Paradox")

The length of your IV dictates how many unique messages you can safely encrypt using the same secret key before you risk repeating an IV. Repeating an IV with the same key is a catastrophic cryptographic failure.

128-bit IV (CBC Mode): Extremely large space (2^128). You can safely generate random IVs for trillions of messages without ever worrying about a duplicate.

96-bit IV (GCM Mode): While $2^{96}$ is still a massive number, the math behind random collisions (the Birthday Paradox) dictates that if you generate purely random 96-bit IVs, you shouldn't encrypt more than roughly (2^32) (about 4.2 billion) messages under a single key.

How to handle the 96-bit limit in GCM:

Because 96 bits is a bit tighter for random generation, cryptographic standards suggest two ways to stay safe:

  • The Counter Approach (Deterministic): Use a fixed 64-bit device ID combined with a 32-bit incrementing counter as your IV. Because it counts up, it will never repeat until the counter rolls over.

  • Key Rotation: If you must use purely random 96-bit IVs, generate a fresh encryption key well before you hit a billion messages.
